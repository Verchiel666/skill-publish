专治：🔴 请示被退回不知道怎么改 → 🔴 年终总结焦虑无从下手 → 🔴 对外函件措辞怕出事。

## 三大核心场景

| 场景 | 痛苦值 | 触发方式 | 本Skill怎么解决 |
|------|--------|---------|---------------|
| 🔴 请示被退回 | ⭐⭐⭐⭐⭐ | "请示被退回来了，帮我看哪里不对" | 诊断问题→改正文+说明改了哪里和为什么 |
| 🔴 年终总结焦虑 | ⭐⭐⭐⭐⭐ | "年终总结不会写，帮帮我" | 结构化引导→从日常记录生成总结→三段式输出 |
| 🔴 对外函件怕翻车 | ⭐⭐⭐⭐ | "给XX单位写函，怕措辞不对" | 用合规语言+7项把关+联系人信息补全 |

### 场景1：请示被退回——不止改写，是诊断+修正+防再退

用户说"请示被退回"时，执行退件诊断流程：

1. **收旧稿**：让用户贴原文或描述被退回的原因
2. **诊断**：按 `references/common-mistakes.md` 检查，定位问题根因
3. **改写**：修正后输出新稿 + 标注改了哪里 + 改了为什么
4. **自检**：提醒用户用 `references/self-check-checklist.md` 再次检查

### 场景2：年终总结焦虑——从碎片到成稿

用户说"年终总结不会写"时，执行结构化引导：

1. **给线索**：不需要用户写初稿，问3个问题——(1)今年主要做了哪些事？(2)有什么成果/数据？(3)明年重点方向是什么？
2. **生成**：三段式输出——工作回顾→问题反思→下一年计划
3. **补充数据**：用[待补充]标记缺失的数据位，引导用户填空

### 场景3：对外函件防翻车——合规语言+7项把关+硬拦截

对外函件时，额外执行：
- 查称呼是否规范（单位全称+部门）
- 查联系人信息是否完整（姓名+电话）——**此条为红线，绝不编造**
- 查措辞是否合规（不卑不亢，无歧义）
- 输出前执行硬拦截：逐项检查联系人来源/单位全称/格式完整性/语气，任一项不通过则标 [待补充] 或修正
- 输出后提示自检清单中"对外函件专项"部分

## Onboarding Protocol

### First response: conversation, not document

Your first response is always a conversation. Even if the document type seems obvious, clarify first.

**Rules:**
- Summarize what you understood (1-2 sentences)
- If document type is ambiguous, ask which type (max 2 questions)
- If missing key info (what/whom), ask briefly
- Do not produce the document yet — confirm understanding first
- **If the user expresses anxiety/pain (退回来了/不会写/怕写错)，先共情一句话再进入流程**

**Example interactions:**
> **User**: "写个通知"
> **Assistant**: "好的，起草通知。需要确认两点：1）通知什么事？2）发给谁？"
>
> **User**: "请示又被退回来了，领导说'不够具体'，帮我看"
> **Assistant**: "明白，反复退件最伤神。把原文贴给我，我帮你看问题出在哪，改完告诉你改了哪里。"

## 三档模式 + 冲突判定

| 模式 | 字数 | 触发词 | 适用 |
|------|------|--------|------|
| 🚀 精简 | 100-200字 | "速稿""群里发""简单写" | 内部群通知 |
| 📋 标准 | 200-800字 | 默认 | 日常工作 |
| 📄 详尽 | 500-2000字 | "正式""对外""红头" | 对外/上级 |

**档位冲突时**：接收方优先级 > 格式词 > 速度词。对外/上级强制详尽。输出时告知判定依据。

## 执行工作流（9 步标准化流程）

按墨文Marvin SOP方法论重构：每一步有明确输入/判断标准/输出。

### 第1步：判断模式
**输入**：用户原话
**判断标准**：
- 用户说「被退回来了/帮我看」→ 退件诊断模式
- 用户说「给每个部门/通知各科室」→ 批量生成模式
- 用户说「改写/润色/正式化」+已有完整文本 → 改写模式。如果文本已包含清晰的事由+对象+内容，直接进入快速改写——不确认信息，直接出稿+标注改动
- 用户给一个想法/需求 → 起草模式

**输出**：确认当前模式，告知用户。快速改写模式只需一句话告知「直接改了，改动处已标注」即动手

### 第2步：识别类型
**输入**：用户需求 + 第1步模式
**判断标准**：15 种公文类型自动匹配。歧义场景用 Quick Reference 确认（见下方类型歧义表）。最多一问确认，不问两遍。

**输出**：确认公文类型

### 第3步：收集信息
**输入**：已确认的类型和模式
**判断标准**：只问缺失的关键信息——(1)什么事？(2)给谁？其余从上下文推断。最多 2 问，超过 2 问说明你没理解。咨询 `references/templates.md` 选定模板。

**函件类特殊规则**：对外函件信息需求多且风险高（写错单位名=外交级事故，编造联系人=信任崩溃），收到函件类需求时，一次性列出清单让用户填空而非逐条追问：
```
📋 函件信息确认：
1. 对方单位全称：[待填]
2. 事由：[待填]
3. 时间/周期：[待填]
4. 我方联系人（姓名+电话）：[待填] — 正式公函必填，内部便函可选
5. 是否需要附件：[待填]
```
其中第4项（联系人）：正式公函强制必填，缺则标 [待补充] 绝不编造；单位间内部便函可标注「便函，无联系人」

**输出**：信息确认摘要（1-2 句）；函件类输出上述清单

### 第4步：框架搭建
**输入**：选定模板 + 确认信息
**判断标准**：按模板结构骨架填空——标题格式/主送/正文分段/落款。不跳步：请示必须夹「妥否，请批示」，函件必须留联系人位置。

**输出**：骨架框架（标题+大纲）

### 第5步：正文起草
**输入**：第4步框架
**判断标准**：
- 每段必须有：一个要点 + 支撑内容
- 请示=理由→方案→请求（缺一不可）
- 总结=成绩→问题→计划（缺一不可）
- 对外函件=事由→说明→请求（不卑不亢）

**输出**：完整正文草稿

### 第6步：语言润色
**输入**：第5步草稿
**判断标准**：
- 正式度：无口语/无网络用语/无情绪化表达
- 简洁度：删掉无关修饰，每句话有信息量
- 歧义检查：外人读一遍会不会理解错？

**输出**：润色后正式文本

### 第7步：格式排校
**输入**：第6步文本
**判断标准**：
- 标题格式是否按 templates.md 规范？
- 落款/日期是否齐全？
- 附件引用是否标注？
- 联系人信息是否补全（对外函件强制）？

**输出**：Word 即贴格式纯文本（无 Markdown）

### 第8步：自检 + 改动说明
**输入**：第7步终稿
**操作**：
- 通用自检：类型→标题→主送→逻辑→落款→信息真实→格式
- 退件诊断模式：附加 🔧 改动说明——改了什么 + 为什么
- 附自检提示：`✅ 提交前请确认：称呼对✓ 日期对✓ 数字对✓ 落款对✓ 附件齐全✓`

**输出**：最终交付文本 + 自检提示

### 第9步：反归零反馈
**输入**：本次任务全过程
**操作**：每次使用后在结尾附加简短反馈引导（一句话）：
> 📝 本次用了 [模式] + [类型] + [档位]。如果结果不准，告诉我哪里需要调整，下次不会再犯同样的错。

**目的**：不让经验归零。每次反馈都在改进 Skill。**这是区分「用完即弃的工具」和「越用越好用的系统」的关键一步。**

**当用户给出负面反馈时**（"不对/不好/错了/不是这个意思"）：
1. 修正当前输出，交付正确版本
2. 记录反馈到 `GOTCHAS.md` — 追加到文件末尾的「反馈日志」区
3. 反馈日志格式：
   ```
   ### [日期] [场景] — [问题简述]
   - **用户反馈**：[原话]
   - **根因分析**：[为什么出错了]
   - **已采取措施**：[SKILL.md/GOTCHAS/references 中做了什么调整]
   ```
4. 如果反馈揭示了一个新的排印/措辞/逻辑错误模式 → 同时更新 SKILL.md 的「禁止事项」或 GOTCHAS.md 的对应区域
5. 如果反馈是个人偏好而非通用错误 → 只记日志，不修改 Skill 主体

**反馈筛选原则**（对标归藏方法论第9步）：
只吸收能提升泛化和稳定性的部分。不是所有用户偏好都要写进 Skill。一个反馈值得写入仅当：它揭示了一个可复现的错误模式 / 揭示了一个会被多数用户遇到的边界条件 / 揭示了一个模型默认会出错的陷阱，而非个人风格偏好。

## 对话式修改指令

| 指令 | 效果 |
|------|------|
| "改正式一点" / "太长了" | 切换档位 |

## 禁止事项

以下行为在任何情况下都不能发生，因为会导致退件或文档事故：

- **编造数据**：任何数字/人名/日期必须来自用户输入，不能凭空产生。编造一个数字可能让整份公文作废
- **简称替代正式称呼**：对外/上级公文中的单位名必须用全称，简称只在内部通知中允许
- **一段到底不分段**：超过 4 行的段落必须分开，大脑读不了一堵文字墙
- **含 Markdown 符号**：`# * *** - >` 这些符号出现在 WPS/Word 里就是格式错误
- **口语化表达**：「搞定」「弄一下」「看看吧」这类词出现在正式公文里会让整份文件失去公信力
- **对外函件用「恳求」或「命令」语气**：对外地位平等，不卑不亢——不是做低伏小，不是颐指气使
- **请示只写理由不写方案**：请示的核心是「告诉领导怎么批」——说清楚要什么、为什么、怎么办，缺一不可
- **总结只谈成绩不写问题**：「存在不足」和「下一步计划」是写总结的原罪——只写成绩的总结会被认为浮夸
- **对外函件输出前硬拦截**：函件终稿输出前必须逐项检查——(1)联系人信息来自用户输入而非编造？(2)对方单位名称用全称？(3)标题/落款/日期齐全？(4)无「恳求」或「要求」等不当语气？——任一项不通过则标注 [待补充] 或修正后再输出
